## 컨텍스트

- 로드맵 문서: @docs/ROADMAP.md

## 작업

대상 Phase: $ARGUMENTS

- `$ARGUMENTS`가 비어있으면 `docs/ROADMAP.md`의 모든 Phase를 검사한다.
- `$ARGUMENTS`에 특정 Phase 번호(예: `4` 또는 `Phase 4`)가 지정되면 해당 Phase만 검사한다.

### 1. 체크 대상 파악

`docs/ROADMAP.md`에서 각 Phase의 "작업 목록" 섹션에 있는 체크박스(`- [ ]` / `- [x]`) 항목을 모두 추출한다. 각 항목은 보통 다음 중 하나를 의미한다:

- 특정 파일/함수의 생성 (예: `src/lib/notion.ts`, `getCourses()`)
- 특정 설정 값 적용 (예: `revalidate = 60`, 환경 변수)
- 특정 UI 동작 (예: 모바일 햄버거 메뉴, 태그 필터링)

### 2. 실제 상태 검증

체크박스 하나마다, 근거 없이 상태를 바꾸지 말고 반드시 코드/설정을 직접 확인한다:

- 파일 생성 여부 → 해당 경로를 `ls` 또는 Read로 확인
- 함수 구현 여부 → `grep`으로 함수 시그니처나 export 여부 확인
- 설정 값 적용 여부 → 파일 내용에서 실제 값 확인 (예: `revalidate = 60`, `generateMetadata` 존재 여부)
- 배포/외부 서비스 관련 항목(Vercel 배포, Notion Integration 연결 등)처럼 코드만으로 확인 불가능한 항목은 자동으로 체크하지 말고 "수동 확인 필요" 항목으로 별도 표시한다.
- 이미 `[x]`로 체크된 항목이 실제로는 구현이 삭제/변경되어 더 이상 유효하지 않다면, 이것도 발견하여 보고한다 (체크 해제 후보).

### 3. 로드맵 갱신

검증 결과를 바탕으로 `docs/ROADMAP.md`의 체크박스를 실제 상태에 맞게 Edit로 수정한다:

- 구현이 확인된 항목: `- [ ]` → `- [x]`
- 구현이 삭제/무효화된 것으로 확인된 항목: `- [x]` → `- [ ]`
- 문서의 다른 서술(Phase 설명, 이유 등)은 사용자가 명시적으로 요청하지 않는 한 변경하지 않는다.

### 4. 결과 보고

다음 형식으로 짧게 요약한다:

- 새로 체크된 항목 목록 (근거가 된 파일/코드 위치 포함)
- 체크 해제된 항목이 있다면 그 이유
- 코드만으로 확인 불가능해 수동 확인이 필요한 항목 목록
- 어느 Phase가 완전히 완료되었는지, 다음으로 진행 가능한 Phase는 무엇인지

### 주의사항

- 추측으로 체크하지 않는다. 파일이 존재하지 않거나 grep 결과가 없으면 미완료로 간주한다.
- 하나의 체크박스 항목이 여러 하위 조건을 담고 있으면(예: "getCourses(), getNotes(), getNote() 세 함수 구현"), 모든 하위 조건이 충족되어야 체크한다.
- `docs/PRD.md`나 `CLAUDE.md`와 로드맵 서술이 충돌하는 경우, 최신 코드 상태를 우선 기준으로 삼되 충돌 사실을 보고에 언급한다.
